# Plan: Custom Dashboard Backend (Block-based Hybrid)

## Goal

Add a custom admin dashboard at `/admin` so the firm can manage every piece of dynamic content themselves:

- All 18 pages' SEO titles, meta descriptions, OG images, and per-section body content
- All 14 service entries (add / edit / remove)
- Site logo, contact info, social links, footer text
- Form submissions (browse, search, export CSV)

The dashboard is built with shadcn/ui + Auth.js + Drizzle + Neon, fully owned by us (no vendor lock-in). The public site's Webflow HTML stays 100% intact — the dashboard only injects content into marked zones, so the look, layout, fonts, animations, and JS-driven interactions are preserved exactly as they are today.

## Why Custom (not Payload, Sanity, or Tina)

The full rationale is in `backend-plan-payload.md` (kept for reference) and the discussion log below. Short version:

- **Tina** is a poor fit — markdown-first, designed for static content. Our site is hardcoded Webflow HTML; converting to markdown changes the look.
- **Payload** has the weakest admin UX (functional but utilitarian, hard to customise deeply).
- **Sanity** is great UX but hosted (vendor lock-in, free-tier limits).
- **Custom dashboard** gives the best UX (full design control), zero lock-in, comparable build time (~5-6 days), and matches the brand perfectly.

The data model is small (18 pages, 14 services, 1 settings, 1 form-submission collection) — a custom dashboard is faster to build than configuring a hosted CMS for this scope.

## Stack

- **Frontend (admin)**: shadcn/ui + Tailwind (only in the `/admin` route group; the public site keeps the original Webflow CSS)
- **Auth**: Auth.js v5 with email magic links (no passwords to manage — admin enters email, gets a one-click login link in their inbox, session lasts 7 days)
- **DB**: Neon Postgres (already provisioned in the Payload plan; same connection strings work)
- **ORM**: Drizzle (light, type-safe, no codegen)
- **Forms**: react-hook-form + zod (for client-side validation) + server actions
- **Rich text editor**: Tiptap (light, beautiful, ~50KB; used for the body block content)
- **File upload**: Browser → signed URL → Vercel Blob (direct upload, server doesn't proxy the file)
- **Hosting**: Vercel (already set up, auto-deploys on push)

The admin and the public site share the same Next.js app. The `/admin` route group has its own root layout with Tailwind. The public site uses the existing Webflow CSS. The shared piece is the data layer (Drizzle schema + the zone-injection in `app/_lib/page.ts`).

## Data Model (Drizzle schema sketch)

Five tables, same shape as the Payload plan but defined in TypeScript via Drizzle instead of auto-generated by Payload:

```ts
// db/schema.ts
export const pages = pgTable('pages', {
  id: serial('id').primaryKey(),
  slug: text('slug').notNull().unique(),     // '/', '/contact', '/services/housing'
  title: text('title').notNull(),             // H1
  metaTitle: text('meta_title').notNull(),
  metaDescription: text('meta_description').notNull(),
  ogImageUrl: text('og_image_url'),           // null = use the default Southwest-og.jpg
  showInNav: boolean('show_in_nav').default(true),
  body: jsonb('body').notNull().default('{}'), // { 'home-hero': { type: 'hero-block', ... }, ... }
  updatedAt: timestamp('updated_at').defaultNow().notNull(),
});

export const services = pgTable('services', {
  id: serial('id').primaryKey(),
  slug: text('slug').notNull().unique(),
  name: text('name').notNull(),
  subtitle: text('subtitle'),                  // '01', '02'… the order number
  cardImageUrl: text('card_image_url'),
  description: text('description').notNull(),
  longDescription: jsonb('long_description').notNull().default('{}'), // Tiptap JSON
  contactFormEnabled: boolean('contact_form_enabled').default(true),
  displayOrder: integer('display_order').default(0),
});

export const siteSettings = pgTable('site_settings', {
  id: serial('id').primaryKey(),              // always = 1, single row
  logoUrl: text('logo_url'),
  companyName: text('company_name').notNull(),
  companyTagline: text('company_tagline'),
  address: text('address'),
  phoneNumbers: jsonb('phone_numbers').notNull().default('[]'),  // ['01392 984 206', ...]
  email: text('email'),
  socialLinks: jsonb('social_links').notNull().default('{}'),    // { instagram, twitter, linkedin, facebook }
  registrationNumber: text('registration_number'),
  registeredOffice: text('registered_office'),
  footerText: text('footer_text'),
  updatedAt: timestamp('updated_at').defaultNow().notNull(),
});

export const contactSubmissions = pgTable('contact_submissions', {
  id: serial('id').primaryKey(),
  name: text('name').notNull(),
  email: text('email').notNull(),
  phone: text('phone'),
  message: text('message').notNull(),
  source: text('source').notNull(),            // 'contact-page' | 'service-page:<slug>'
  submittedAt: timestamp('submitted_at').defaultNow().notNull(),
});

export const adminUsers = pgTable('admin_users', {
  id: text('id').primaryKey(),                // matches Auth.js user id
  email: text('email').notNull().unique(),
  name: text('name'),
  // Auth.js handles its own session table; we just track the email allowlist
});
```

The single admin user is `iraselrony@gmail.com` (the user we've been working with). The `admin_users` table acts as the email allowlist — Auth.js can sign in any user but only those in `admin_users` can access `/admin/*` routes.

## Editable Zones (the block-based hybrid)

Same approach as the Payload plan. The Webflow HTML stays untouched, and the body extract injects CMS content into marked zones.

1. **Zone markers in the body extract** — wrap each editable section in `<div data-payload-zone="<zone-id>">…</div>`. The zone-id is stable (e.g. `home-hero`, `service-hero`).
2. **Zone map per page** in `app/_lib/zones.ts`:

   ```ts
   export const ZONES: Record<string, string[]> = {
     '/': ['home-hero', 'about', 'service-cards', 'contact-cta'],
     '/contact': ['contact-intro', 'contact-form-copy'],
     '/services/<slug>': ['service-hero', 'service-body'],
     // … 4 main pages + 14 service pages
   };
   ```

3. **Inject CMS content** in `extractBodyInner` (~20 lines): for each `<div data-payload-zone="<id>">`, fetch the matching block from the page's `body` JSON column and replace the inner HTML. If no CMS data exists, fall back to the original Webflow content.
4. **Render** as today via `dangerouslySetInnerHTML`. The Webflow CSS still controls the layout. Only the marked zones get CMS content.

### Block types stored in the `body` JSON column

The body is a `jsonb` column with shape:

```ts
{
  "home-hero":     { type: "hero-block",     image: "...", heading: "...", subheading: "..." },
  "about":         { type: "rich-text-block", content: { /* Tiptap JSON */ } },
  "service-cards": { type: "service-cards-block", heading: "...", serviceSlugs: ["housing", "retail", ...] },
  "contact-cta":   { type: "cta-block",       heading: "...", subheading: "...", buttonText: "...", buttonUrl: "..." }
}
```

The admin UI renders a different form per block type (Tiptap editor for rich text, file upload + text fields for hero, etc.).

## File Upload Flow

Browser → `/api/admin/upload/sign` → server returns a Vercel Blob signed upload URL → browser PUTs the file directly to Blob → server saves the resulting Blob URL into the relevant row.

This means the file never touches our server (Vercel function doesn't proxy it). Faster uploads, no Vercel function timeout issues for large files.

## Out of scope (defer to v2)

- **React component refactor** — per-section real components. The block-based hybrid is "good enough"; the React refactor is cleaner long-term but takes 2-4 weeks. Defer until the firm wants design flexibility.
- **Multi-user with roles** — single admin user for now. Add per-user content ownership in v2.
- **WYSIWYG preview in admin** — the admin shows a list of form fields, not a live page preview. The "Preview" link opens the public site in a new tab. Could add a real preview pane in v2.
- **Page-level access control** — single admin can edit all pages. Add per-page ACLs in v2.
- **i18n / multilingual** — site is English-only.
- **Form-to-Slack / form-to-CRM integrations** — the form persists + emails. No webhook integrations.
- **Cookie consent banner** — GDPR thing, separate task.
- **Domain cutover** (DNS `www.southwestplanningconsultancy.co.uk` → Vercel) — separate task.
- **Production rename** (`southwest-planning-clone` → `southwest-planning-consultancy`) — separate task.

## Critical Tradeoff

A custom dashboard requires more code than configuring Payload. The payoff:

- **No plugin layer** to fight against (Payload's React admin is a black box once you go beyond CSS tweaks)
- **No vendor lock-in** (no Sanity)
- **No markdown conversion** (no Tina)
- **Full design control** (matches the firm's brand exactly)
- **The admin is a Next.js app, just like the public site** — same mental model, same deployment, same auth

For a 5-6 day build that gets us all the way there, the custom dashboard is the right answer.
